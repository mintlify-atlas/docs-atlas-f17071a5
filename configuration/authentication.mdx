Scrapyd supports HTTP basic authentication to restrict access to the web UI and API endpoints. When enabled, all requests must include valid credentials.

<Note>
  Authentication was added in Scrapyd 1.3.0
</Note>

## Enabling Authentication

To enable basic authentication, set both `username` and `password` in your configuration file:

```ini scrapyd.conf
[scrapyd]
username = admin
password = secretpassword
```

Both settings must be non-empty for authentication to be enabled. If either is empty, authentication is disabled.

<Warning>
  Usernames cannot contain colons (`:`) as per [RFC 2617](https://www.rfc-editor.org/rfc/rfc2617#section-2). Scrapyd will raise an `InvalidUsernameError` if you attempt to use a username containing a colon.
</Warning>

## Using Environment Variables

For better security in production environments, use environment variables instead of storing credentials in configuration files:

```bash
export SCRAPYD_USERNAME=admin
export SCRAPYD_PASSWORD=secretpassword
scrapyd
```

Environment variables take precedence over configuration file settings.

<Tip>
  Store credentials in environment variables or secrets management systems rather than committing them to version control.
</Tip>

## Making Authenticated Requests

### Using curl

When authentication is enabled, include credentials with the `-u` flag:

```bash
curl -u admin:secretpassword http://localhost:6800/daemonstatus.json
```

### Scheduling a Spider

```bash
curl -u admin:secretpassword http://localhost:6800/schedule.json \
  -d project=myproject \
  -d spider=myspider
```

### Uploading a Project

```bash
curl -u admin:secretpassword http://localhost:6800/addversion.json \
  -F project=myproject \
  -F version=r23 \
  -F egg=@myproject.egg
```

### Using Python Requests

```python
import requests
from requests.auth import HTTPBasicAuth

auth = HTTPBasicAuth('admin', 'secretpassword')

# Check daemon status
response = requests.get(
    'http://localhost:6800/daemonstatus.json',
    auth=auth
)
print(response.json())

# Schedule a spider
response = requests.post(
    'http://localhost:6800/schedule.json',
    auth=auth,
    data={
        'project': 'myproject',
        'spider': 'myspider'
    }
)
print(response.json())
```

### Using scrapyd-client

Include credentials in the target URL:

```bash
scrapyd-deploy --target http://admin:secretpassword@localhost:6800
```

Or configure in `scrapy.cfg`:

```ini scrapy.cfg
[deploy:production]
url = http://admin:secretpassword@localhost:6800/
```

## Authentication Implementation

Scrapyd implements HTTP basic authentication using Twisted's authentication framework. The implementation is in `scrapyd/basicauth.py:44-58`:

```python
def wrap_resource(resource, config):
    username = os.getenv("SCRAPYD_USERNAME") or config.get("username", "")
    password = os.getenv("SCRAPYD_PASSWORD") or config.get("password", "")
    
    # RFC 2617: usernames cannot contain colons
    if ":" in username:
        raise InvalidUsernameError

    if username and password:
        log.info("Basic authentication enabled")
        return HTTPAuthSessionWrapper(
            Portal(PublicHTMLRealm(resource), [StringCredentialsChecker(username, password)]),
            [BasicCredentialFactory(b"Scrapyd")],
        )

    log.info("Basic authentication disabled as either `username` or `password` is unset")
    return resource
```

The authentication flow:

1. Environment variables are checked first (`SCRAPYD_USERNAME`, `SCRAPYD_PASSWORD`)
2. If not set, configuration file values are used
3. If both username and password are non-empty, authentication is enabled
4. All HTTP requests are wrapped with `HTTPAuthSessionWrapper`
5. Credentials are verified using `StringCredentialsChecker`

## Security Considerations

<Warning>
  Basic authentication sends credentials in Base64 encoding, which is **not encrypted**. Always use HTTPS when exposing Scrapyd to untrusted networks.
</Warning>

### Best Practices

<AccordionGroup>
  <Accordion title="Use HTTPS in production">
    Basic authentication credentials are Base64-encoded but not encrypted. Deploy Scrapyd behind a reverse proxy (nginx, Apache) with HTTPS enabled:
    
    ```nginx
    server {
        listen 443 ssl;
        server_name scrapyd.example.com;
        
        ssl_certificate /path/to/cert.pem;
        ssl_certificate_key /path/to/key.pem;
        
        location / {
            proxy_pass http://localhost:6800;
            proxy_set_header Host $host;
        }
    }
    ```
  </Accordion>
  
  <Accordion title="Use environment variables for credentials">
    Never commit credentials to version control. Use environment variables or secrets management:
    
    ```bash
    # In systemd service file
    [Service]
    Environment="SCRAPYD_USERNAME=admin"
    Environment="SCRAPYD_PASSWORD=secretpassword"
    ExecStart=/usr/local/bin/scrapyd
    ```
    
    Or use a secrets management system like HashiCorp Vault, AWS Secrets Manager, or Kubernetes secrets.
  </Accordion>
  
  <Accordion title="Use strong passwords">
    Generate strong, random passwords:
    
    ```bash
    # Generate a secure random password
    openssl rand -base64 32
    ```
  </Accordion>
  
  <Accordion title="Restrict network access">
    Combine authentication with network-level security:
    
    ```ini
    [scrapyd]
    # Only listen on localhost
    bind_address = 127.0.0.1
    username = admin
    password = secretpassword
    ```
    
    Then use SSH tunneling or VPN to access Scrapyd remotely.
  </Accordion>
  
  <Accordion title="Use Unix sockets for local access">
    For local-only access, use Unix sockets instead of TCP:
    
    ```ini
    [scrapyd]
    bind_address =
    unix_socket_path = /var/run/scrapyd/web.socket
    ```
    
    The socket file permissions (660) provide access control at the filesystem level.
  </Accordion>
</AccordionGroup>

## Troubleshooting

### Authentication Not Working

If authentication doesn't seem to be enabled, check the Scrapyd logs:

```
2024-03-05 10:00:00+0000 [-] Basic authentication enabled
```

or

```
2024-03-05 10:00:00+0000 [-] Basic authentication disabled as either `username` or `password` is unset
```

Common issues:
- One of `username` or `password` is empty
- Configuration file is not being read (check file locations)
- Environment variables are overriding configuration with empty values

### Invalid Username Error

If you see `InvalidUsernameError`, your username contains a colon:

```
scrapyd.exceptions.InvalidUsernameError
```

Choose a username without colons:

```ini
# Bad
username = admin:user

# Good
username = admin_user
```

### 401 Unauthorized Responses

If you receive `401 Unauthorized` responses:

```bash
curl -v http://localhost:6800/daemonstatus.json
# < HTTP/1.1 401 Unauthorized
# < WWW-Authenticate: Basic realm="Scrapyd"
```

Make sure to include credentials:

```bash
curl -u admin:secretpassword http://localhost:6800/daemonstatus.json
```
